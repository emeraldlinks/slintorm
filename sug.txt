What's genuinely pointless for Mongo:
ALTER TABLE-style DDL, obviously — there's no schema to alter.
Column type/constraint enforcement — NOT NULL, UNIQUE, CHECK, foreign keys — none of that exists at the Mongo layer. SlintORM would have to enforce all of it in application code before the write, and it already does (that's the "SlintORM enforces shape, Mongo just stores JSON" point from before). Migrations that exist purely to codify constraint changes have nothing to attach to.
Anything your migrator does around @omitmigrate (excluding a field from DDL) is meaningless when there's no DDL to exclude it from.
What can still be legitimately useful, even schemaless:
Indexes. Mongo absolutely has indexes (createIndex), and @index/@unique annotations should translate to db.collection.createIndex(...) calls — that's real, meaningful "migration" work for Mongo, just via the Mongo driver's index API instead of SQL DDL.
Data migrations — backfilling a new field across existing documents, renaming a field, restructuring nested documents. This is the category of "migration" that matters most for schemaless stores in practice (every real-world Mongo app ends up writing these by hand eventually), and it's a legitimate thing an ORM's migration system could help with — but based on what we've seen in dbAdapter.js and the docs' "second-class citizen" framing, I'd bet SlintORM's migrator doesn't currently attempt this for Mongo; it's a much harder problem (needs to iterate/transform live documents, not just diff a schema) and would be new work, not a free extension of the SQL migrator.
So: schema/constraint migrations — yes, correctly useless for Mongo, nothing to fix. Index management — should exist and be genuinely useful, worth checking if it does. Data-shape migrations — the actually valuable thing for Mongo users, almost certainly not implemented, and would be a meaningfully different feature (not a Mongo branch of the existing migrator) if you ever wanted it.
